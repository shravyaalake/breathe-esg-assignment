# MODEL.md

## Overview

The Multi-Tenant ESG Emissions Data Ingestion & Tracking Platform is designed to collect emissions-related data from multiple enterprise source systems, normalize it into a common ESG data model, classify emissions into Scope 1, Scope 2, and Scope 3 categories, and maintain full traceability back to the original source data.

The model emphasizes:

* Multi-tenancy
* Auditability
* Source traceability
* Data normalization
* ESG scope classification
* Extensibility for additional source systems

---

# Core Domain Model

## Tenant

Represents an organization using the platform.

### Attributes

| Field      | Description             |
| ---------- | ----------------------- |
| id         | Unique identifier       |
| name       | Organization name       |
| industry   | Industry classification |
| created_at | Creation timestamp      |

### Example

```text
ABC Manufacturing Ltd
Industry: Automotive
```

### Purpose

All ESG data belongs to a tenant, ensuring logical separation between organizations.

---

## Facility

Represents a physical site or operational location belonging to a tenant.

### Attributes

| Field         | Description         |
| ------------- | ------------------- |
| id            | Unique identifier   |
| tenant        | Associated tenant   |
| facility_code | Facility identifier |
| name          | Facility name       |
| created_at    | Creation timestamp  |

### Example

```text
BLR01 - Bengaluru Plant
PUN01 - Pune Plant
CHE01 - Chennai Plant
```

### Relationship

```text
Tenant
  └── Multiple Facilities
```

## Upload Batch

Represents a CSV ingestion event.

### Attributes

| Field         | Description             |
| ------------- | ----------------------- |
| id            | Unique identifier       |
| tenant        | Associated tenant       |
| source_system | Source of uploaded file |
| file_name     | Uploaded file           |
| uploaded_at   | Upload timestamp        |

---

## Emission Record

Represents a normalized ESG activity record.

### Attributes

| Field           | Description              |
| --------------- | ------------------------ |
| id              | Unique identifier        |
| tenant          | Associated tenant        |
| facility        | Related facility         |
| source_system   | Originating source       |
| activity_data   | Raw activity value       |
| unit            | Measurement unit         |
| emission_factor | Applied factor           |
| emissions_co2e  | Calculated emissions     |
| scope           | ESG scope classification |
| created_at      | Record timestamp         |

### Standardized Structure

```text
Activity Data
Unit
Emission Factor
CO₂e Emissions
Scope
Facility
Tenant
Source System
```

### Purpose

Serves as the normalized source of truth for ESG reporting and analytics.

# ESG Scope Classification

The platform classifies emissions according to the GHG Protocol framework.

## Scope 1

Direct emissions generated by owned or controlled operations.

Examples:

```text
Diesel Consumption
Natural Gas Usage
Fuel Combustion
```

---

## Scope 2

Indirect emissions from purchased electricity.

Examples:

```text
Electricity Consumption
Grid Power Usage
Utility Bills
```

---

## Scope 3

Indirect emissions occurring throughout the value chain.

Examples:

```text
Supplier Procurement
Purchased Goods
Business Travel
Employee Travel
```

# Data Normalization Model

Different source systems provide data in varying formats.

The ingestion layer transforms all inputs into a common ESG structure.

### Transformation Flow

```text
Source CSV
      ↓
Validation
      ↓
Parsing
      ↓
Unit Normalization
      ↓
Scope Classification
      ↓
Emission Calculation
      ↓
Normalized ESG Record
```

### Benefits

* Consistent reporting
* Comparable analytics
* Reduced source-specific complexity
* Easier auditing and compliance

---

# Emission Calculation Model

Emissions are calculated using the general formula:

```text
Emissions (kgCO₂e)
=
Activity Data × Emission Factor
```

### Electricity

```text
kWh × Grid Emission Factor
```

### Travel

```text
Distance × Travel Emission Factor
```

### Procurement

```text
Spend × Spend-Based Emission Factor
```

# Source Traceability

Every emission record retains metadata linking it to its origin.

Stored metadata includes:

```text
Tenant
Facility
Source System
Upload Timestamp
Original File
```

### Traceability Path

```text
Emission Record
      ↓
Upload Batch
      ↓
Original CSV
      ↓
Source System
```

This ensures transparency and supports ESG audit requirements.

# Multi-Tenant Architecture

The platform follows a shared-database, tenant-scoped architecture.

### Isolation Strategy

```text
Tenant
   ├── Facilities
   ├── Source Systems
   ├── Upload Batches
   └── Emission Records
```

All business entities are associated with a tenant, enabling future expansion to support multiple organizations while maintaining logical data separation.

---

# Design Principles

The data model was designed around the following goals:

1. Multi-tenant support
2. ESG-focused normalization
3. Source traceability
4. Auditability
5. Scope classification
6. Extensible ingestion pipelines
7. Consistent reporting structure
8. Deployment simplicity

The model prioritizes transparency, maintainability, and future scalability while remaining simple enough for rapid implementation and demonstration.
