# 1. Multi-Tenancy Design

## Ambiguity

The assignment requires multi-tenancy but does not explicitly define:

* database isolation strategy
* tenant authentication model
* whether tenants share infrastructure
* how tenant access should be enforced

## Decision Taken

Implemented **logical multi-tenancy** using tenant relationships.

Most business entities belong to a tenant:

```text
Tenant
Facility
SourceSystem
IngestionBatch
RawRecord
NormalizedActivity
```

Example:

```text
ABC Manufacturing Ltd
```

owns:

```text
Facilities
Source Systems
Uploads
Activities
```

## Why

This approach is:

* simple to implement
* easy to explain
* scalable for prototype scope
* sufficient for demonstrating tenant separation

It also maps naturally to relational databases.

Alternative approaches considered:

### Separate databases per tenant

Rejected because:

* too heavy for prototype scope
* increases operational complexity
* unnecessary for assignment

### Separate schema per tenant

Rejected because:

* adds migration complexity
* over-engineered for a prototype

## What I Would Ask the PM

* Are tenants expected to share infrastructure?
* Is strict data isolation required?
* Are enterprise customers expected to require database-level isolation?

# 2. Scope 1 / 2 / 3 Categorization

## Ambiguity

The assignment requires:

```text
Scope 1 / Scope 2 / Scope 3 categorization
```

but does not define:

* exact mapping rules
* emissions factor source
* required calculation accuracy

## Decision Taken

Implemented simplified classification rules.

### Scope 1

Fuel combustion records:

```text
Diesel fuel usage
```

mapped to:

```text
SCOPE_1
```

because emissions are directly generated by operations.

### Scope 2

Model support exists but sample workflow is minimal.

Would normally include:

```text
Purchased electricity
```

mapped to:

```text
SCOPE_2
```

### Scope 3

Procurement activities mapped to:

```text
SCOPE_3
```

because they represent indirect value-chain emissions.

## Why

This gives:

* meaningful ESG categorization
* realistic business interpretation
* working review dashboard data

without requiring a full sustainability framework implementation.

## What I Would Ask the PM

* Which emissions framework should be used?

  * GHG Protocol?
  * ISO?
  * internal ESG standard?
* Should categorization rules be configurable?

# 3. Source-of-Truth Tracking

## Ambiguity

The assignment requires:

```text
source-of-truth tracking
```

but does not define what level of traceability is expected.

Questions include:

* file-level?
* row-level?
* version-level?
* edit history?

## Decision Taken

Implemented traceability using:

```text
tenant
source_system
ingestion_batch
raw_record
timestamps
review status
```

Each normalized activity can be traced back to:

```text
tenant
facility
source system
CSV upload
raw row
```

## Why

This provides meaningful traceability while keeping implementation complexity reasonable.

A reviewer can answer:

```text
Where did this row come from?
Which upload created it?
Which tenant owns it?
Which system produced it?
```

## Not Implemented

Full immutable row history.

Example:

```text
old value → new value
who changed it
reason for change
```

was intentionally simplified.

## What I Would Ask the PM

* Is regulatory-grade auditability required?
* Must edits be immutable?
* How long should history be retained?

# 4. Unit Normalization Strategy

## Ambiguity

The assignment requires:

```text
unit normalization
```

but does not define:

* supported units
* conversion precision
* canonical unit system

## Decision Taken

Implemented lightweight normalization.

Examples:

```text
Litre
litres
Liter
L
```

normalized to:

```text
L
```

Supported sample-style normalization:

```text
L
KG
EA
```

The model separates:

```text
original quantity + original unit
normalized quantity + normalized unit
```

## Why

This supports:

* cleaner dashboard rendering
* consistent analytics
* future CO2e calculations

without building a large conversion framework.

## Not Implemented

Examples intentionally excluded:

```text
kg → ton
mile → km
currency conversion
regional conversion logic
```

## What I Would Ask the PM

* Which canonical unit system should be used?
* Are regional conversion rules required?
* How accurate must conversions be?

# 5. CSV Upload as Primary Ingestion Method

## Ambiguity

The assignment references real-world source systems:

```text
SAP
Utility Portal
Travel Platform
```

but does not specify whether ingestion must be:

```text
API-based
file-based
batch-based
streaming
```

## Decision Taken

Implemented:

```text
CSV upload
```

as the ingestion mechanism.

Supported source systems:

```text
SAP Procurement Export
Utility Portal Export
Travel Platform Export
```

through uploaded CSV files.

## Why

CSV uploads are:

* realistic for enterprise exports
* easy to validate
* quick to demonstrate
* sufficient for assignment goals

Many enterprise ESG workflows begin as exported CSVs before API integration.

## What I Would Ask the PM

* Are APIs expected later?
* Should ingestion be scheduled?
* Are near real-time updates needed?

# 6. Review Workflow

## Ambiguity

The assignment references review and audit concepts but does not define workflow behavior.

Questions include:

* who can approve?
* can records be edited?
* can approvals be reversed?

## Decision Taken

Implemented a simple review lifecycle:

```text
PENDING
APPROVED
REJECTED
LOCKED
```

User actions:

```text
Approve
Reject
Lock
```

are available from Activity Review.

## Why

This is enough to demonstrate:

* workflow state transitions
* reviewer actions
* audit logging
* dashboard interaction

without introducing RBAC or approval chains.

## What I Would Ask the PM

* Who is allowed to approve?
* Are approvals reversible?
* Are comments mandatory?
* Is dual approval required?

# 7. CO2e Calculation Scope

## Ambiguity

The assignment mentions emissions tracking but does not define:

* calculation methodology
* emissions factor source
* precision requirements

## Decision Taken

Implemented basic CO2e calculation for supported fuel combustion records.

Example:

```text
100L diesel → calculated CO2e
```

Procurement activities are normalized but may not always contain CO2e.

## Why

Procurement emissions require more complex assumptions:

```text
supplier factors
spend-based factors
regional factors
material-specific factors
```

Implementing a realistic Scope 3 engine was considered out of assignment scope.

## What I Would Ask the PM

* Which emissions factor library should be used?
* Should calculations be transparent to reviewers?
* Must factor versioning exist?

# 8. Persistence Strategy on Free Hosting

## Ambiguity

The assignment requires deployment but does not specify hosting persistence expectations.

Free Render instances may reset local storage.

## Decision Taken

Implemented:

```text
seed_demo_data
```

to repopulate demo tenant, facility, and source-system master data after redeployment.

The frontend provides upload guidance:

```text
Tenant ID = 1
Source System ID = 1
```

for demonstration purposes.

## Why

This makes the deployed application usable for reviewers even if demo data resets after redeployment.

## What I Would Ask the PM

* Should demo data persist permanently?
* Is production-grade persistence expected in evaluation?

# 9. Chosen Source Subset

The assignment references three source domains:

```text
SAP Procurement
Utility Portal
Travel Platform
```

Only a representative subset of each was modeled.

Implemented examples:

### SAP

Handled:

```text
fuel procurement
material procurement
basic quantities
vendor information
```

Ignored:

```text
ERP purchase lifecycle
invoice matching
SAP-specific metadata
```

### Utility Portal

Handled:

```text
utility-style activity ingestion
basic normalization
```

Ignored:

```text
billing cycles
meter hierarchy
regional tariffs
```

### Travel Platform

Handled:

```text
travel activity representation
CSV ingestion shape
```

Ignored:

```text
booking systems
airline integrations
hotel emissions logic
distance estimation
```

## Why

The objective was to demonstrate architecture and ingestion flow, not fully reproduce enterprise systems.

Reason:

This assignment appears to optimize for:

```text
architecture thinking
data modeling
tradeoff awareness
product reasoning
working deployment
```

rather than production completeness.

The goal was to demonstrate good engineering judgment and a scalable foundation while staying realistic for implementation time.
